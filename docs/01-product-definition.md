# Product Definition

## 1. 제품 개요

새 제품은 "여러 프로젝트를 메신저처럼 오가며, 프로젝트별로 독립된 터미널 워크스페이스를 운영하는 데스크톱 터미널"이다.

이 제품은 단순한 탭 터미널이 아니다. 핵심은 다음 두 가지다.

- 프로젝트 중심 탐색
- 레이아웃 중심 작업

즉 사용자는 "셸 창 하나"를 여는 것이 아니라, 프로젝트 단위의 작업 공간에 들어가서 탭과 pane을 배치하며 여러 CLI 세션을 유지한다.

## 2. 해결하려는 문제

기존 터미널 앱과 일반적인 탭 UI는 다음 문제를 충분히 해결하지 못한다.

- 프로젝트가 많아질수록 어떤 세션이 어떤 프로젝트에 속하는지 기억하기 어렵다.
- 프로젝트를 전환할 때 작업 문맥이 끊기기 쉽다.
- 같은 프로젝트 내에서 여러 세션을 열고 split으로 배치하다 보면 구조가 빠르게 흐트러진다.
- AI CLI를 쓰는 개발 흐름에서는 한 프로젝트 안에서 일반 셸, 실행 서버, 테스트, 로그, AI 작업 세션이 동시에 필요하다.
- 기존 앱은 탭은 강하지만 프로젝트 계층이 약하거나, 반대로 프로젝트 관리는 되지만 레이아웃이 약하다.

## 3. 제품 목표

### 3.1 핵심 목표

- 프로젝트를 메신저 사이드바처럼 빠르게 전환할 수 있어야 한다.
- 프로젝트별로 상단 워크스페이스 탭과 내부 split-pane 구성을 영속적으로 유지해야 한다.
- 세션은 프로젝트 전환이나 UI 이동과 무관하게 백그라운드에서 살아 있어야 한다.
- 탭과 pane의 구조가 커져도 사용자가 현재 어디서 무엇을 보고 있는지 쉽게 파악할 수 있어야 한다.
- 특정 AI CLI, 특정 셸, 특정 provider에 묶이지 않는 범용 터미널 플랫폼이어야 한다.

### 3.2 비기능 목표

- 입력 지연이 적고, 대량 출력 시에도 UI가 버벅이지 않아야 한다.
- 앱 재실행 후에도 마지막 상태를 안정적으로 복원해야 한다.
- 레이아웃과 세션 상태가 분리되어 있어 UI 변경이 세션 안정성을 해치지 않아야 한다.
- Windows를 우선 지원하되, 구조상 macOS/Linux 확장을 막지 않아야 한다.

## 4. 목표 사용자

### 4.1 1차 사용자

- 여러 코드베이스를 동시에 다루는 개인 개발자
- AI CLI와 일반 셸을 번갈아 사용하는 개발자
- 서버 실행, 로그 확인, 테스트, 코드 작업을 여러 세션으로 분리하는 사용자

### 4.2 2차 사용자

- 프리랜서 또는 컨설턴트처럼 클라이언트 프로젝트를 자주 오가는 사용자
- 여러 저장소와 브랜치를 병렬로 다루는 사용자
- 장기적으로 팀 협업용 세션 공유나 에이전트 운영을 염두에 둔 고급 사용자

## 5. 핵심 사용자 시나리오

### 5.1 프로젝트 전환 시나리오

1. 사용자는 좌측 프로젝트 목록에서 `Project A`를 선택한다.
2. 해당 프로젝트의 마지막 워크스페이스 상태가 복원된다.
3. 상단에는 이 프로젝트에서 열어 둔 탭들이 표시된다.
4. 각 탭 안에는 split-pane 트리와 활성 세션이 그대로 살아 있다.
5. 사용자가 `Project B`로 이동해도 `Project A`의 세션은 종료되지 않는다.

### 5.2 다중 세션 시나리오

한 프로젝트 안에서 사용자는 다음을 동시에 운영할 수 있다.

- 메인 셸
- 개발 서버
- 테스트 감시 세션
- 로그 tail 세션
- AI CLI 세션

이 세션들은 여러 탭에 분산될 수도 있고, 하나의 탭 안에서 split-pane으로 동시에 볼 수도 있다.

### 5.3 레이아웃 중심 시나리오

1. 사용자는 새 탭을 만든다.
2. 탭 안의 루트 pane을 세로 split한다.
3. 왼쪽 pane은 메인 셸, 오른쪽 pane은 테스트 세션을 연다.
4. 오른쪽 pane 내부를 stack으로 전환하여 로그 세션과 테스트 세션을 번갈아 본다.
5. 해당 탭 구조는 자동 저장된다.

## 6. 제품 범위

### 6.1 포함 범위

- 프로젝트 등록, 삭제, 정렬, 검색
- 프로젝트별 워크스페이스 유지
- 상단 탭
- pane split, stack, focus, close, move
- PTY 세션 생성, 종료, 재시작, resize
- 앱 종료 후 상태 복원
- 사용자 설정: 폰트, 테마, 기본 셸, 기본 시작 디렉토리, 키 바인딩 일부
- command palette와 context menu
- 네이티브 데스크톱 패키징

### 6.2 초기 제외 범위

현재 설계 문서에서는 다음을 명시적으로 제외한다.

- 멀티에이전트 오케스트레이션
- 에이전트 간 메시징
- Claude Teams 호환 레이어
- 자동 승인 로직
- 파일 잠금/큐 관리
- 원격 제어 서버
- 클라우드 동기화

이 항목들은 제품 확장 영역이지, V1의 핵심 구조 정의에는 포함하지 않는다.

## 7. UX 원칙

### 7.1 프로젝트 우선

사용자의 정신 모델은 "지금 어떤 프로젝트에 들어와 있는가"가 먼저다. 탭과 pane은 프로젝트 컨텍스트 안에서만 해석된다.

### 7.2 세션은 살아 있고, 뷰는 바뀐다

세션은 런타임 리소스이고, 탭과 pane은 세션을 보여주는 뷰다. 뷰를 닫는 것과 세션을 종료하는 것은 동일하지 않다.

### 7.3 레이아웃은 사용자의 자산

레이아웃은 일회성 UI 상태가 아니라 작업 문맥이다. 탭 구조, split 비율, stack 구성, 활성 탭은 모두 복원 가능해야 한다.

### 7.4 provider 중립

제품은 특정 AI 도구를 위한 특수 UI가 아니라, 어떤 CLI든 다룰 수 있는 범용 터미널이어야 한다. 제품 내부 개념에서 `Claude`, `Codex`, `Gemini` 같은 특정 이름은 핵심 도메인 용어가 되지 않는다.

## 8. 성공 기준

- 프로젝트 전환 시 사용자가 작업 문맥을 잃지 않는다.
- 탭과 pane 구조가 많은 상태에서도 조작과 복원이 일관된다.
- 세션이 UI와 분리되어 안정적으로 유지된다.
- UI 구조와 PTY 코어가 분리되어 추후 에이전트/오케스트레이션 기능을 무리 없이 얹을 수 있다.
